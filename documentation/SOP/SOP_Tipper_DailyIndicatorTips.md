## 1. Purpose

The Tipper script generates daily threat indicator tips for HTOC partner organizations. It pulls recently observed indicators from ThreatConnect, cross-references them against OpDiv observation data, filters to high-confidence malicious indicators (VirusTotal score > 10), enriches them with Shodan and VirusTotal data, and outputs a partner-bucketed Excel workbook â€” one tab per partner â€” to the shared Tippers directory.

The output gives each partner a targeted, enriched view of threat indicators actively observed in their environment within the last 24 hours that meet a minimum malicious threshold.

---

## 2. Scope

This procedure applies to HTOC analysts and data engineers who run, monitor, or troubleshoot the daily Tipper process. It covers the full pipeline from ThreatConnect query through Excel output, as well as the batch launcher and its logging behavior.

## 6. Step-by-Step Logic

### Step 1 â€” Initialize ThreatConnect

`initialize_threatconnect()` reads credentials from `config.json`, instantiates the `ThreatConnect` SDK object, and creates a global `RequestObject` pre-configured with HTTP `GET` and owner `HTOC Org`. On failure, the script prints an error and exits with code 1.

---

### Step 2 â€” Load OpDiv Observation CSVs

`get_file_paths()` generates file paths for the last **3 days** using UTC dates, then filters to only paths that exist on disk. `load_observed_data()` reads each file and concatenates them into a single DataFrame (`observed_data_df`). Missing files are skipped with an error message.

---

### Step 3 â€” Query ThreatConnect

`query_observed_src()` queries the ThreatConnect `/v3/indicators` endpoint using TQL for:

- `ownerName EQ "HTOC Org"`
- `typeName IN` (Address, EmailAddress, File, Host, URL, ASN, CIDR, Email Subject, Hashtag, Mutex, Registry Key, Stripped URL, User Agent)
- `lastObserved >= {3 days ago}T00:00:00Z`

Fields retrieved: `tags`, `observations`. Results are normalized, deduplicated by `indicator`, and filtered to indicators with `lastObserved` within the 3-day window. The result is stored as `observed_src`.

If `observed_src` is empty, the script exits early.

---

### Step 4 â€” Process Recent Tags

`process_recent_tags()` identifies partner-linked indicators by:

1. Iterating over `observed_src` and exploding `tags.data` for each indicator.
2. Normalizing the tag name `VA CSOC CTS Splunk` â†’ `VA Splunk API`.
3. Keeping only indicators that have at least one tag containing `"API"` (case-insensitive) â€” these are the partner Splunk API tags.
4. Inner-joining with `observed_data_df` on `summary` + `OpDiv`.
5. Filtering to indicators where both `lastObserved` and `obs_date` fall within the **last 24 hours**.
6. Extracting `partner` from the tag name (strips `" Splunk API"` suffix).
7. Aggregating partner counts and names per indicator into `partner_count` and `Partners` columns.
8. Deduplicating to one row per indicator.

If no matches are found, the script exits early.

---

### Step 5 â€” Exclude SOAR-Created Indicators

`excludeSoarData()` calls the ThreatConnect `/v3/indicators/{indicator}?fields=attributes` endpoint for each indicator in `recent_tags`. It:

- Collects attribute metadata including `createdBy.lastName`.
- **Excludes** any indicator whose attributes were created by `lastName == 'SOAR'` â€” these are automated SOAR-generated entries, not analyst-confirmed threats.
- Indicators with **no attributes at all** are retained (they are not considered SOAR-generated).
- HTTP 400 errors are silently suppressed (unsupported indicator types).

This step also triggers the **enrichment process** (Step 6) for the filtered set.

---

### Step 6 â€” Enrich with VirusTotal and Shodan

For each remaining indicator, a POST request is sent to:
```
/v3/indicators/{encoded_value}/enrich?type=Shodan&type=VirusTotalV3
```

Enrichment data is flattened and merged back onto `filtered_recent_tags`. Shodan fields extracted:

| Field | Column |
|---|---|
| `hostNames` | `shodan_hostNames` |
| `domains` | `shodan_domains` |
| `tags` | `shodan_tags` |
| `country` | `shodan_country` |
| `city` | `shodan_city` |
| `isp` | `shodan_isp` |
| `asn` | `shodan_asn` |
| `org` | `shodan_org` |
| `openPorts` | `shodan_openPorts` |

VirusTotal field extracted: `vtMaliciousCount`

Shodan 400 errors for unsupported indicator types are silently suppressed.

---

### Step 7 â€” Filter by VT Malicious Score

Only indicators with `vtMaliciousCount > 10` are retained. This threshold ensures the output contains only high-confidence malicious indicators. Indicators without VirusTotal data or with a low score are dropped.

---

### Step 8 â€” Finalize and Rename Columns

The enrichment fields are extracted via `extract_enrichment()` and joined back. Columns are renamed to analyst-friendly names:

| Internal Name | Output Column |
|---|---|
| `indicator` | `Indicator` |
| `vtMaliciousCount` | `Malicious Score/Count` |
| `obs_date` | `Observation Date` |
| `shodan_asn` | `ASN` |
| `rating` | `ThreatAssessRating` |
| `confidence` | `ThreatAssessConfidence` |
| `shodan_city` | `City` |
| `shodan_country` | `Country` |
| `data.legacyLink` | `ThreatConnect Link` |
| `partners` | `Partners` |

Missing values are filled with `"unknown"`. Duplicates are dropped.

---

### Step 9 â€” Bucket by Partner

`all_partners` is derived from the `Partners` column (comma-separated values). A `partner_buckets` dictionary is built mapping each partner name to its subset of `filtered_recent_tags` using whole-word regex matching.

If no partners are found, the script exits without writing output.

---

### Step 10 â€” Save Excel Workbook

`save_to_excel()` writes one Excel worksheet per partner to:
```
\\10.1.4.22\data\HTOC\HTOC Reports\Tippers\tippers_by_partner_{YYYYMMDD}.xlsx
```

Each sheet:
- Is named using the partner name (sanitized to alphanumeric/underscore/hyphen, max 31 characters per Excel limit).
- Has **autofilter** enabled on all columns.
- Has the **header row frozen** (freeze panes at row 1).
- Has **auto-sized column widths** (capped at 50 characters).
- Strips timezone info from datetime columns to ensure Excel compatibility.

## 7. Batch launcher (`run_tipper.bat`)

The batch file **`run_tipper.bat`** is the supported operator entry point: it upgrades key packages, runs **`main.py`**, and writes structured logs next to the script.

To run without the batch file, install **`pandas pytz numpy requests xlsxwriter openpyxl`** (and any missing imports **`main.py`** requires), **`cd`** to the **`Tipper`** folder so **`utils/`** resolves, then run **`python main.py`** using the interpreter your team standardizes on.

### Execution sequence:

1. **Writes `[` to `TipperRunLogs.json`** â€” initializes a fresh JSON array, overwriting the previous run's log.
2. **Logs a `Start` event** with a UTC timestamp.
3. **Runs `pip install --upgrade --quiet --user`** for: `pandas pytz numpy requests xlsxwriter openpyxl`. Output appended to `TipperRunOutput.log`. If pip fails (non-zero exit code), logs a `PipInstallFailed` event and exits.
4. **Runs `main.py`**. All stdout/stderr is appended to `TipperRunOutput.log`.
5. **Logs an `End` event** with a UTC timestamp and the script's exit code.
6. **Closes the JSON array** with `]`.

### Log files:

| File | Location | Contents |
|---|---|---|
| `TipperRunLogs.json` | `Tipper\TipperRunLogs.json` | Structured JSON log: Start / PipInstallFailed / End events with timestamps and exit codes |
| `TipperRunOutput.log` | `Tipper\TipperRunOutput.log` | Full stdout/stderr from pip install and `main.py` (appended each run) |

> **Important:** `TipperRunLogs.json` is **overwritten** at the start of every run, so it only reflects the most recent execution. `TipperRunOutput.log` is **appended** and will grow over time â€” archive or truncate periodically.

## 9. Troubleshooting

| Symptom | Likely Cause | Resolution |
|---|---|---|
| `TipperRunLogs.json` shows `PipInstallFailed` | Network/proxy blocked pip download | Run the pip install command manually from a terminal to confirm connectivity; packages may already be installed â€” check if `main.py` can be run directly |
| `[ERROR] Failed to initialize ThreatConnect` | Invalid or expired API credentials | Update `utils\config.json` with current credentials; verify ThreatConnect is reachable from the host where you ran **`run_tipper.bat`** / **`main.py`** |
| `[DEBUG] observed_src is empty` | No indicators found in TC for the 3-day window | Check ThreatConnect for indicator activity; verify the query owner (`HTOC Org`) and lookback are correct |
| `[DEBUG] recent_tags is empty after processing` | No indicators matched to partner API tags in the last 24 hours | Verify OpDiv CSVs for the target dates exist; check that partner API tags are being applied in ThreatConnect |
| `[DEBUG] filtered_recent_tags is empty` | All indicators were either SOAR-created or had VT score â‰¤ 10 | Normal on low-activity days; review SOAR exclusion criteria or lower the VT threshold if needed |
| `[DEBUG] all_partners is empty` | No partners in `Partners` column after filtering | Confirms no qualifying indicators for any partner today; no output file will be written |
| Excel file missing for today | Script exited early (empty data at any stage) | Review `TipperRunOutput.log` for the `[DEBUG]` message that triggered the early exit |
| `TipperRunOutput.log` is very large | Log is append-only and accumulates across runs | Truncate or archive the file periodically; do not delete â€” it is the primary debug record |
| Shodan data missing for some indicators | Shodan does not support that indicator type (e.g., URL, Hash) | Expected behavior; Shodan 400 errors for unsupported types are suppressed; Shodan fields will be `unknown` for those rows |
